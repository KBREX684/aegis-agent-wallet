# Aegis Agent 钱包项目框架（通用版）

## 0. 文档目的
本文件用于统一 Aegis 的产品定义、合规边界、技术架构与实施路径，作为对内研发和对外路演的共同基线。

## 1. 项目定位
Aegis 是一个“人类在回路（Human-in-the-Loop）”的 Agent 支付审批层。

Aegis 允许 AI Agent 自主发起支付请求，但每一笔交易最终执行都必须经过自然人确认。Aegis 不替代支付宝、微信、美团等支付/交易平台，不托管用户资金，而是作为 Agent 与人之间的支付审批网关。

核心价值：
- 把人类从“填单、跳转、重复确认”的高频操作中解放出来。
- 将支付操作降维为“审一下、按一下”。
- 在满足监管底线前提下释放 Agent 执行效率。

## 2. 核心原则
1. 人类最终签署
- 任何资金流动必须由自然人最终确认。
- 逐笔确认与预授权并行，但预授权本身也是一次明确的人类决策。

2. Agent 无权自主支付
- Agent 不持有支付私钥，不具备资金签名能力。
- Agent 仅能提交结构化支付请求。

3. 三级额度管控
- 第一级：可用额度分配（未分配部分为保护余额，Agent 不可见）。
- 第二级：消费规则（白名单商户、单笔上限、日累计上限、允许时段）。
- 第三级：执行模式（逐笔签署或预授权窗口）。

4. 全流程可审计
- 请求、审批、执行、回调、拒绝原因码均留痕。
- 支持交易哈希复算与审计事件追踪。

## 3. 合规与资金架构
Aegis 采用“不碰资金”的审批层架构：
- 平台不接收、不存储、不划转用户资金。
- 资金始终留存在用户的数字人民币运营机构钱包。
- Aegis 仅处理审批信息流与授权指令流。

在产品语义上：
- “保护余额 / 可用额度”是逻辑视图，不是平台内真实资金容器。
- 平台中的额度变更是权限分配，不是资金沉淀。

## 4. 产品能力边界
Aegis 负责：
- Agent 接入与身份验证
- 请求校验与风控分流
- 人类审批与预授权管理
- 审计记录与回调分发

Aegis 不负责：
- 替代银行/支付机构做清算
- 持有或托管数字人民币私钥
- 绕过用户最终授权进行隐式扣款

## 5. 技术架构
三层结构：
1. Agent 层
- 通过 SDK/Connector 发起 JSON 请求（payee、amount、purpose、expires_at、nonce、signature）。

2. 网关层（Aegis Core）
- 鉴权、防重放、规则校验、额度校验、预授权命中、待签队列、执行编排、回调通知、审计记录。

3. 用户端（Web/App）
- 审批签署、额度分配、规则配置、预授权管理、消费与审计查看。

## 6. 用户与 Agent 接入闭环
1. 用户注册并登录 Aegis。
2. 用户在 Aegis 生成“一键安装 Aegis Connector”链接。
3. 用户把安装链接发送给目标 Agent。
4. Agent 安装后生成绑定链接并回传 Aegis。
5. 用户点击绑定链接，回到 Aegis 完成登录与确认绑定。
6. 绑定完成后，Agent 才能代表该用户发起支付请求。

## 7. 运行闭环（交易链路）
1. Agent 发起请求。
2. 网关执行签名、nonce、时效、规则、额度校验。
3. 命中预授权则进入自动执行；否则进入待签队列。
4. 用户端进行生物认证/签署确认。
5. 网关执行并回调 Agent，写入消费记录与审计事件。

## 8. 通用实施路线（不绑定具体时间）
阶段 A：最小模拟 MVP
- Web 前端 + Python 网关 + 模拟 Agent。
- 实现请求、签署、执行、回调、审计闭环。

阶段 B：成熟 MVP
- 完成“保护余额/可用额度”模型。
- 完成 Agent 安装链接与绑定闭环。
- 完成 API 额度采购演示主场景。

阶段 C：最终展示 MVP
- 从网页优先演进到 App 优先。
- 接入生物认证能力与推送通知增强。
- 优化交互与路演稳定性（弱网、隧道、异常恢复）。

阶段 D：最初版真实产品
- 对接数字人民币真实执行链路（按合作条件选择银行接口、子钱包推送、鸿蒙 Payment Kit 等）。
- 强化多租户、权限体系、合规审计导出、企业审批流。

## 9. 数据与审计框架
核心数据域：
- 用户与认证资料
- Agent 与绑定关系
- 配额与规则策略
- 支付请求与审批记录
- 消费记录与交易哈希
- 系统审计事件流

审计要求：
- 全事件可回放
- 原因码可追踪（如 RULE_BLOCKED、QUOTA_INSUFFICIENT、EXPIRED）
- 关键字段哈希可复算

## 10. 风险与应对
1. 接口开放不确定
- 采用“模拟执行 -> 跳转执行 -> 深度直连”渐进方案。

2. 预授权被误解为自动扣款
- 明确为“用户显式授权窗口内执行”，并保留事后通知与撤销能力。

3. 推送不可靠
- 推送 + 轮询 + 超时取消三重机制。

4. 平台生态封闭
- P0 聚焦 API 采购等可闭环场景，避免重依赖外部平台开放 API。

## 11. 商业模式
1. 面向 Agent 开发者：审批网关 API 服务费（按调用量/层级计费）。
2. 面向企业：SaaS 订阅费（Agent 数量、管理员席位、审计报表、审批流能力）。
3. 中长期：在用户授权与合规前提下提供消费分析增值服务。
4. 远期：与数字人民币运营机构探索导流合作。

## 12. 验收标准（通用）
1. 能稳定完成“请求 -> 审批 -> 执行 -> 回调 -> 审计”闭环。
2. 规则拦截、额度不足、重放攻击等异常可明确识别。
3. 消费记录可按 Agent 检索，交易哈希可复算。
4. 绑定流程可重复演示，误绑定与重复绑定可防护。
5. 在本地与公网隧道场景下均可演示。
